We're still getting bold and italics wrong

1 July, 2024

I see it far too often -- on the internet, many rich text editors use <strong> and <em> in place of <b> and <i> by default. What many don't realise is that when it comes to semantic HTML, these aren't intended to be interchangeable and actually have different semantic meaning.

Let's go through each one.

<strong> and <em>

I think of <strong> and <em> as semantic tags. When they're applied to some text, they add semantic meaning to the text.

By default, text in a strong tag is bolded, and text in an em tag is italicised. Keep in mind, however that this can be changed with CSS.

Using <strong> tags

According to the HTML spec, the <strong> tag:

represents strong importance for its contents.

When you apply <strong> to words, you're saying that those words are serious, important, or urgent in some way.

The most common use would be to mark something as a warning, or important.

<strong>Warning</strong>: This area is dangerous. <strong>Avoid travelling through this area.</strong>
<strong>Check your answers before submitting your details.</strong>

This means the only time you use it is to mark something as semantically important.

The HTML spec actually goes on to say:

The relative level of importance of a piece of content is given by its number of ancestor strong elements; each strong element increases the importance of its contents.

This means that nested <strong> tags actually increase the level of importance.

Using <em> tags

According to the HTML spec, the <em> tag:

represents stress emphasis of its contents.

When you apply <em> to words, you're adding stress emphasis to those words, changing the meaning of the text. Have a look at the following example:

This is your one.

The meaning of this sentence will change depending on which words are stressed:

  • This is your one - This one (as opposed to the other one) is yours
  • This is your one - I'm certain that this particular one is indeed yours
  • This is your one - This one (out of others) is yours in particular
  • This is your one - You only get one of these, and this is your one of these

In this case, you'd use <em> to mark this specific stress emphasis. While <i> marks these words visually, <em> marks it semantically in the HTML. This means that <em> is actually a integral part of the content as it alone explains this stress meaning. If for whatever reason the CSS stylesheet isn't working and therefore not showing italics, the underlying HTML would still have the correct stress emphasis.

Similar to <strong>, the HTML spec says:

The level of emphasis that a particular piece of content has is given by its number of ancestor em elements.

This means that you can express different levels of stress, depending on the number of <em> ancestor tags there are. For example:

<em>This <em>is</em> your one!</em>

This sentence shows the speaker is certain this one is yours, and is emphatically fighting to have that position heard.

<b> and <i>

I think of <b> and <i> as presentational tags. When applied, text is visually marked apart from other text, either for utilitarian purposes or to indicate some different form or quality of text.

Using <b> tags

According to the HTML spec, the <b> tag:

represents a span of text to which attention is being drawn for utilitarian purposes without conveying any extra importance and with no implication of an alternate voice or mood, such as key words in a document abstract, product names in a review, actionable words in interactive text-driven software, or an article lede.

Put simply, you use <b> to make text look bold so that it's easier to use. For example, you could make key words bold so they're easier to identify.

Note that this bolded text doesn't indicate alternate voice or mood, and it doesn't impart any additional meaning.

Using <i> tags

According to the HTML spec, the <i> tag:

represents a span of text in an alternate voice or mood, or otherwise offset from the normal prose in a manner indicating a different quality of text, such as a taxonomic designation, a technical term, an idiomatic phrase from another language, a thought, or a ship name in Western texts.

The <i> tag has a lot going on, but the main purpose is to show that something is being said in a different voice, mood, or manner.

You could use it to show that some text is a thought or some other manner of speaking.

<i>I hope this doesn't go badly</i>, Ada prayed.

You'd also use it to mark phrases in another language, technical terms, taxonomic designations, and very uniquely, for the name of ships in Western texts.

"I'll see you later, <i lang="fr">ma cherie</i>," promised Wilson. Wilson quickly boarded <i>The Washington</i>. The future of <i class="taxonomy">homo sapiens</i> was relying on their work being successful.

Note that you should be using the <dfn> tag rather than <i> if you're defining a term for the first time. For more information, see The dfn element - HTML5: Edition for Web Authors.

Effect on web accessiblilty

My understanding is that most major screen readers don't actually signify the semantics of these tags.

Why? We might find an answer in the NVDA screen reader, which signified it for a brief time. It was reported as being extremely unpopular among users and resulted in a lot of complaints. They remarked that emphasis was over-used in the wild.

When every rich text editor uses <strong> and <em> as bold and italics as default, it's no wonder it's overused.

What next?

It's a huge shame that these tags have been misused. There's an opportunity to create richer text and richer meaning, but misuse has ironically made the web less accessible.

This problem highlights a larger issue: Everyone can easily write online, but few know online writing conventions. So how can we ensure that the average person writing for the web is writing semantically?

How do we solve an issue like this? I'm not sure. If all web users and rich text editors were to tag everything correctly from this point forward, we'd still have a slew of existing content to fix.

Maybe in future we could use AI as a solution:

  • GPTs could check and fix misused tags - this doesn't solve the issue of rich text editors misusing the tags and creating even more un-semantic content.
  • We could create AI screen readers - rather than relying on semantic HTML to read out a page, it could visually parse a page's text and meaning, without accessing semantic tagging at all. It would then rely on visual and textual clues to discern and signify meaning.

It's all conjecture. I don't know what the future holds or how the semantic web might change. This is what we have currently, and until it changes I'll continue to tag things as correctly as I can -- we owe it to everyone to make the web as accessible as possible.
